# Reliability & engineering-hardening — the trust substrate

Status: **§1–§4 SHIPPED 2026-07-12**, extended 2026-07-19 by the corpus-generality
hardening (below); **open: §5 (process limits — threat-model-conditional) and §6
(maintainability extraction — gated on characterization tests)**. Sourced from an
external implementation review (2026-07-12, recorded in NOTES.md) that read the code and
ran focused experiments. These are NOT commercial-parity *features* (see
commercial-parity.md) and NOT QA *gates* (see qa-methodology.md) — they are the
engineering substrate that makes the project's own central claims ("validated",
"byte-reproducible", "config records reviewed judgment") true rather than aspirational.

**Framing.** The pipeline's value proposition is *trustworthiness through independence*:
evidence → recorded judgment → deterministic build → independent QA → proofread. The
review found several load-bearing promises in that chain unenforced or leaky. None was a
routine failure in the human-in-the-loop workflow — they were latent holes that turn a
regression, a stale artifact, or a silently-ignored judgment into a
*believable-but-wrong* result: exactly the failure mode the architecture exists to
prevent. The single highest-value review finding — the page-aligned fidelity gate — is
not here; it shipped as gate 25 ([qa-methodology.md §3](qa-methodology.md)).

## Shipped record (§1–§4, 2026-07-12)

Implementation narrative in NOTES.md ("Reliability roadmap Tier 0 + Tier 1 SHIPPED");
key decisions and divergences preserved here:

- **§1 Config integrity.** The doctrine at stake: book.yaml is the record of reviewed
  judgment, so a value that silently does nothing — or a placeholder that ships — is a
  judgment believed-recorded but not applied. Shipped: shared semantic validator in
  `load_config(require_complete=)` (recursive FILL-ME-IN rejection naming the dotted key
  path, `schema_version`, structural page-range validation) + the `pdf2epub validate`
  subcommand; the seven dead config fields removed (setting one now errors via
  `_check_keys`), except `include_ncx` which was *implemented* instead (with
  stale-`toc.ncx` removal). Deliberately NOT a Pydantic rewrite — the hand parser's
  per-key errors are an asset.
- **§2 Transactional builds + provenance.** Shipped as single-commit promotion: ALL
  fallible work (epubcheck, input recheck, manifest generate + write) precedes the ONE
  atomic `os.replace` to `{slug}.epub`; the manifest is an atomically-written sidecar.
  `provenance.py` records book.yaml hash of the EXACT parsed bytes + source-PDF
  path/hash (snapshotted before processing, rechecked before promotion), tri-state
  epubcheck, tool/dep versions, git rev + dirty flag, release epoch. `pdf2epub verify`
  checks EPUB↔manifest hash AND input drift-or-absence. **Five rounds of implementation
  review (#1–#5) refined this**: a two-rename-with-rollback design was dropped after the
  rollback logic proved to add more failure modes than it closed; the residual (a
  process-kill between two same-dir renames) is `verify`-detected, not silently
  impossible — true joint atomicity would need directory indirection, which conflicts
  with the git-tracked fixed EPUB path.
- **§3 Package identity & revision metadata.** Shipped: persistent
  `metadata.identifier` (validated UUID/urn, generated by `init`, recorded in config —
  never slug-derived; the BoK arabic variant gets a distinct id) and
  `metadata.released` → `dcterms:modified` (an explicit release epoch, never the print
  year, never wall-clock). The `pages.*` folio cross-check shipped OFF (noisy).
- **§4 Hermetic tests + CI.** Shipped: sibling-repo import removed from the test suite,
  Chrome skip env-gated on *launch* failure (not just absence), pytest markers/tiers,
  hashed `requirements.lock`, ruff (E4/E7/E9/F/I), portable GitHub Actions CI.

**2026-07-19 follow-on — generality hardening** (NOTES.md "the manual disciplines
became infrastructure"): two load-bearing MANUAL disciplines became mechanical as the
corpus doubled — (a) **gate 24 fails on a missing `qa_assertions.yaml`** (authored `[]`
= recorded "no fixes yet"; absence = process hole; `init` scaffolds, never clobbers);
(b) **`pdf2epub corpus`** rebuilds every tracked config (`books/*/book*.yaml` minus
drafts — membership IS the convention, no second list to sync), byte-compares the
shipped EPUBs (byte-reproducible builds make a byte change a behavior change), runs QA,
prints the matrix + per-rule counter deltas vs `books/corpus_baseline.json` (baseline
entries keyed by config sha + PDF sha, so a judgment edit reads "inputs changed", never
as a rule regression); QA sidecars keyed by epub stem (fixes last-run-wins between the
two BoK configs); per-rule telemetry (`space-rule-*` counters) via a pure-observer
`tally` proven byte-identical. CI's corpus job gates on build+QA verdicts, never
`--strict` (byte-compare needs this machine's fonts). **Baseline discipline (review #90,
2026-07-21): re-seed the baseline in the same commit as any pipeline-code change**, or
later corpus runs misattribute the accumulated drift.

## §5 External-process resource limits — OPEN (priority: low; threat-model-conditional)

**Problem (scoped).** Three subprocess sites run unbounded (no `timeout=`, unbounded
`capture_output`): poppler `pdftotext` (`extract/__init__.py`), java epubcheck
(`core/qa_epubcheck.py`), Calibre `ebook-convert` (`kindle.py`). **Correction to the
review:** the Chrome/CDP render path it also flagged is *already* well-bounded — output
to `DEVNULL`, every CDP op deadline-timed (`cdp.py`), render clipped to 600×≤3000 CSS px
(`visual.py`), force-killed on exit. Leave it alone.

**Why low.** The workflow is "a human drops a *trusted* PDF"; a hostile/malformed PDF
wedging poppler is not today's threat model. This is hardening for an untrusted-input
future (batch service, public intake), not a current-workflow correctness gap.

**Design.** Add `timeout=` (generous, per-tool) and bounded output capture to the three
`subprocess.run` sites, with a clear `SystemExit`/skip on timeout. Consider a
page-count/file-size ceiling at extract when untrusted intake becomes real. Do this
*when* the threat model changes; noted here so it isn't rediscovered.

**Integration**: the three call sites + a shared `_run(cmd, timeout=…)` helper.

## §6 Concentration / maintainability — OPEN (cross-cutting; gate on characterization tests)

The code is thoughtful, heavily commented, and unusually test-rich (provenance headers
and stale-judgment detection are strengths). The risk is *concentration*:
`flowbuilder.py` (largest and still growing with each conversion's semantic-block
work), the emitter, the config parser, and the QA runner's `run_qa` orchestration
function. Future table/RTL/OCR work threading through these risks destabilizing prose
conversion.

**Not a rewrite.** First strengthen characterization tests around the seams, *then*
extract behind stable interfaces: furniture/page preprocessing, footnote detection,
semantic-block classification, paragraph joining, anchor placement, and — highest
leverage for future QA work — **individual QA gates behind a common gate interface**
(registry of `name → callable(ctx) -> (ok, lines)`), which makes new gates additive.
The 2026-07-19 corpus snapshots (byte-compare + counter deltas across eleven configs)
are the characterization net this was waiting for; sequence the extraction *alongside*
the first big new content feature, never as a standalone big-bang.

## Non-goals

- Rewriting the config parser or flowbuilder wholesale (incremental extraction only).
- Bounding the CDP path (already bounded — see §5).
- A provenance manifest that embeds wall-clock or non-reproducible fields
  (byte-reproducibility is non-negotiable; the manifest uses source hashes + an
  explicit release epoch).
